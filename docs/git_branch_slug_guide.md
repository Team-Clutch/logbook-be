# 개요
이 문서는 이 프로젝트 리포지토리에서 이루어지는 `git` 브랜치 또는 푸시의 종류를 한정하는 `enum`과 같은 맥락의 용어를 안내하는 문서이다.

***

# 슬러그 종류
- `chore`: 아주 작은 작업이나, 사소한 변경점
- `docs`: `markdown`과 같은 문서 추가
- `feat`: 기능 업데이트(예시: API 추가)
- `design`: 아키텍처에 필요한 ERD나 `mermaid`, 쿼리문을 정리하는 내용을 포함

## 브랜치는 언제 수행해야하는가?
| 조건 | 설명 |
| ---- | ---- |
| 한 가지 목적만 달성함 | 브랜치 목표가 끝남 |
| 실행/확인이 가능함 | 최소한의 로컬에서 동작 |
| 관련 문서까지 반영함 | 코드만 있고 문서가 비어있지 않음 |
| 다음 작업이 이 결과를 기반으로 진행 가능 | 다른 사람이 가져다가 사용할 수 있음 |

> 이 브랜치가 맡은 한 가지 일은 이제 충분히 끝났다.

### 개인적인 가이드(추천받음)
나는 설계를 깊게 하는 스타일이라 브랜치도 너무 크게 잡으면 위험하다는 의견.

| 규칙 | 이유 |
| ---- | ---- |
| 브랜치 하나당 커밋 목표를 문장 하나로 말할 수 있어야 함 | **범위 통제** |
| 브랜치 작업 기간은 1~2일 이내로 | **설계 마비 방지** |
| 한 브랜치에서 환경 수정 + API 구현 + DB 설계 같이 하지 않기 | **리뷰와 회고가 어려워짐** |

> 예시
- 좋은 목표: `Python 버전 기준 통일`
- 좋은 목표: `프로필 mock endpoint 추가`
- 안 좋은 목표: `백엔드 초기 셋업 전체`
